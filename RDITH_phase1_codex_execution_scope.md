# RDITH Phase 1 Execution Scope for Codex

## Purpose of this document

This document defines the exact implementation scope that Codex should follow for the current stage of the project.

The current project stage is **not** full ROI selection training yet. The current stage is:

```text
Raw CSI or generated Doppler heatmap
+ user 6DoF stream
→ RDITH residual heatmap
→ residual RF motion blobs
→ global motion-intent features
→ visualizations that show the difference from the original heatmap
```

At this stage, **ROI definitions are not required**.

ROI definitions are only needed in a later stage when we perform ROI-specific feature pooling or train an ROI selection model. For now, Codex should make sure the repository can run without `roi_path`.

---

# 1. Required runtime inputs

The only required runtime observation inputs are:

## 1.1 Raw CSI or generated heatmap

The pipeline should support at least one of the following:

```text
Option A: raw CSI data
Option B: precomputed heatmap generated by the existing CSI heatmap generator
```

If raw CSI is provided, the pipeline may call or wrap the existing heatmap generator.

If a completed heatmap is provided, the RDITH pipeline should start from that heatmap.

Accepted heatmap types may include:

```text
ToF-Doppler
AoA-ToF-Doppler
range-Doppler-like heatmap
```

The exact format can be `.npz`, `.npy`, or another project-defined format, but the loader must be documented and tested.

## 1.2 User 6DoF stream

The user 6DoF stream should include, per timestamp:

```json
{
  "timestamp": 0.0,
  "position_world": [0.0, 1.6, 0.0],
  "quaternion_xyzw": [0.0, 0.0, 0.0, 1.0]
}
```

The pipeline should use this to estimate:

```text
head linear velocity
head angular velocity
6DoF-explained local motion
```

# 2. Static system configuration is allowed but should not be treated as extra sensing input

The following files or values are allowed as static configuration:

```text
config.json
calibration.json
heatmap metadata
antenna layout metadata
RF-to-world alignment metadata
```

These are **not extra runtime sensing data**. They are static system priors, similar to camera intrinsics.

Codex should avoid describing these as required observation inputs.

The system claim should remain:

```text
Runtime observation input = Raw CSI or generated heatmap + user 6DoF
```

Static configuration may include:

```text
center frequency
wavelength
heatmap bin definitions
Doppler bin definitions
ToF/range bin definitions
RF origin in world frame
RF-to-world rotation
thresholds
smoothing parameters
visualization parameters
```

# 3. What should be implemented now

## 3.1 Heatmap loading layer

Implement or verify a loader that can read the original generated heatmap.

Expected output of the loader:

```python
HeatmapFrameSequence:
    timestamps: np.ndarray              # shape [T]
    energy: np.ndarray                  # shape depends on heatmap type
    doppler_bins: np.ndarray            # shape [D]
    tof_or_range_bins: Optional[np.ndarray]
    aoa_bins: Optional[np.ndarray]
    metadata: dict
```

For `ToF-Doppler`, expected energy shape may be:

```text
[T, R, D]
```

For `AoA-ToF-Doppler`, expected energy shape may be:

```text
[T, A, R, D]
```

The loader should not require ROI definitions.

---

## 3.2 Pose loading and velocity estimation

Implement or verify pose loading from JSON.

Required functions:

```python
load_pose_stream(path: str) -> PoseStream
```

```python
estimate_head_motion(pose_stream: PoseStream) -> HeadMotionStream
```

Definitions:

```python
PoseStream:
    timestamps: np.ndarray              # [T]
    positions_world: np.ndarray         # [T, 3]
    quaternions_xyzw: np.ndarray        # [T, 4]
```

```python
HeadMotionStream:
    timestamps: np.ndarray              # [T]
    linear_velocity_world: np.ndarray   # [T, 3]
    angular_velocity_world: np.ndarray  # [T, 3]
    forward_world: np.ndarray           # [T, 3]
```

Use finite differences for velocities. Smooth if necessary, but keep smoothing configurable.

---

## 3.3 Synchronization

Implement timestamp alignment between heatmap frames and 6DoF poses.

Required function:

```python
sync_heatmap_and_pose(
    heatmap: HeatmapFrameSequence,
    pose: PoseStream,
    method: str = "nearest"
) -> SyncedFrameSequence
```

Expected behavior:

```text
For each heatmap frame, find the closest pose frame.
If timestamp distance exceeds a configurable tolerance, warn or drop the frame.
```

This must be tested.

---

## 3.4 Residual Doppler computation

Implement the Phase 1 RDITH residual computation.

The goal is not yet perfect physical tomography. The immediate goal is to remove motion already explainable by user 6DoF and expose residual RF motion.

Required function:

```python
compute_rdith_residual_heatmap(
    heatmap: HeatmapFrameSequence,
    head_motion: HeadMotionStream,
    config: dict,
    calibration: Optional[dict] = None
) -> RDITHFrameSequence
```

Expected output:

```python
RDITHFrameSequence:
    timestamps: np.ndarray
    original_energy: np.ndarray
    residual_energy: np.ndarray
    residual_score: np.ndarray
    doppler_bins: np.ndarray
    metadata: dict
```

Minimum acceptable implementation:

1. Compute a frame-level expected Doppler component from 6DoF.
2. Suppress or downweight Doppler energy that is consistent with 6DoF-explained motion.
3. Preserve energy that is inconsistent with current head motion.

A practical approximation is acceptable for Phase 1. For example:

```text
For each frame:
    estimate head speed and turn rate
    estimate expected Doppler band caused by head motion
    reduce heatmap energy inside that expected band
    keep energy outside that band as residual energy
```

The output must clearly separate:

```text
original heatmap energy
residual heatmap energy
removed or explained energy
```

Do not require ROI definitions here.

---

## 3.5 RF residual blob extraction

Implement sparse blob extraction from the residual heatmap.

Required function:

```python
extract_residual_blobs(
    rdith: RDITHFrameSequence,
    config: dict
) -> list[ResidualBlobFrame]
```

Expected output per frame:

```python
ResidualBlobFrame:
    timestamp: float
    blobs: list[ResidualBlob]
```

```python
ResidualBlob:
    blob_id: int
    centroid_index: list[float]
    centroid_physical: Optional[list[float]]
    peak_energy: float
    total_energy: float
    doppler_centroid: float
    doppler_bandwidth: float
    entropy: float
    confidence: float
```

Minimum implementation:

```text
threshold residual heatmap
find connected components or local peaks
compute centroid, energy, Doppler centroid, Doppler bandwidth, entropy
```

If physical/world coordinates cannot be reliably computed yet, keep `centroid_physical = null` and document it. This is acceptable for Phase 1.

---

## 3.6 Global intent feature extraction

Implement global features that summarize the residual heatmap without ROI definitions.

Required function:

```python
extract_global_intent_features(
    rdith: RDITHFrameSequence,
    blobs: list[ResidualBlobFrame],
    head_motion: HeadMotionStream,
    config: dict
) -> GlobalIntentFeatureSequence
```

Expected features per frame:

```python
GlobalIntentFeatures:
    timestamp: float
    total_residual_energy: float
    residual_energy_ratio: float
    doppler_centroid: float
    doppler_bandwidth: float
    doppler_entropy: float
    num_residual_blobs: int
    max_blob_energy: float
    temporal_growth: float
    head_speed: float
    head_angular_speed: float
    residual_to_head_motion_ratio: float
```

Definitions:

```text
total_residual_energy:
    sum of residual heatmap energy

residual_energy_ratio:
    sum(residual_energy) / sum(original_energy)

doppler_centroid:
    energy-weighted mean Doppler value

doppler_bandwidth:
    energy-weighted Doppler standard deviation

doppler_entropy:
    entropy of normalized Doppler energy distribution

num_residual_blobs:
    number of detected residual motion blobs

max_blob_energy:
    total energy of strongest residual blob

temporal_growth:
    residual energy at current frame minus previous frame

residual_to_head_motion_ratio:
    residual energy divided by head motion magnitude plus epsilon
```

These features are enough to show that we are doing more than original heatmap generation.

# 4. What should not be required in Phase 1

The following should not be required to run the Phase 1 pipeline:

```text
roi.json
ROI center
ROI bounding box
visibility score
occlusion score
frustum culling result
ML training labels
future gaze labels
```

Those belong to Phase 2.

If the current code requires `roi_path` for the main pipeline, Codex should change it so that:

```text
roi_path is optional
ROI feature extraction is skipped when roi_path is absent
RDITH heatmap, blobs, global features, and visualizations still run
```

The command should produce:

```text
output/phase1_demo/global_intent_features.csv
output/phase1_demo/summary.json

```

# 9. Expected output summary

After running Phase 1, users should be able to say:

```text
We input CSI-derived Doppler heatmaps and user 6DoF.
We estimate which RF motion is explained by head movement.
We suppress the explained part.
We keep residual Doppler energy.
We extract residual motion blobs.
We compute global motion-intent features.
We visualize original heatmap vs residual heatmap, explained energy, blobs, and feature timelines.
```

This is enough to demonstrate progress beyond a standard Doppler heatmap generator.

# 10. Phase 2 is explicitly out of scope for now

Do not make Phase 1 depend on:

```text
ROI definitions
ROI labels
ROI bounding boxes
frustum culling
occlusion culling
ROI selection model training
```

Those are future steps.

Phase 2 will be:

```text
RDITH residual heatmap
+ candidate ROI definitions from the rendering engine
→ RF-ROIAlign
→ ROI-specific feature vectors
→ ROI selection ML model
```

But that is not required for the current milestone.
