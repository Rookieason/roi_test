# RDITH ROI Selection Implementation Plan

## 0. Purpose of this document

This document is written for another AI agent or developer who will continue the coding work.

The goal is to extend the existing CSI heatmap generation code into a real-time ROI-selection feature pipeline for a VR user. The existing code is treated as a standard CSI-based Doppler / ToF-Doppler / AoA-ToF-Doppler heatmap generator. The new work should not replace that generator unless absolutely necessary. The new work should add a layer that converts CSI heatmaps plus user 6DoF into explainable ROI-oriented features.

The proposed method name is:

**RDITH: Residual Doppler Intent Tomographic Heatmap**

The core idea is:

> Use the existing Doppler heatmap as the RF sensing front end, subtract the motion already explainable by the user's HMD 6DoF, then extract ROI-aligned physical-intent features for ML-based ROI selection.

## 1. Hard input assumption and stop condition

### 1.1 Inputs expected by this plan

This plan assumes the implementation has access to either Option A or Option B.

### Option A: Raw CSI + 6DoF

Required inputs:

1. Raw CSI data of the environment.
2. Timestamped 6DoF of the user / HMD.
3. Candidate ROI information from the existing VR pipeline.
4. Calibration metadata that maps RF coordinates and VR/world coordinates.

The existing code can generate heatmaps from raw CSI.

### Option B: Completed heatmap + 6DoF

Required inputs:

1. Completed heatmap generated by the existing heatmap generator.
2. Timestamped 6DoF of the user / HMD.
3. Candidate ROI information from the existing VR pipeline.
4. Axis metadata of the heatmap, such as ToF axis, Doppler axis, AoA axis if available, sampling rate, center frequency, and timestamp mapping.
5. Calibration metadata that maps heatmap cells to VR/world coordinates or at least to user-relative coordinates.

### 1.2 Immediate stop condition

If the implementation does **not** have either raw CSI or completed heatmap, stop.

If the implementation has heatmap but does **not** have heatmap axis metadata, stop and ask for it.

If the implementation has 6DoF but no timestamp alignment between 6DoF and CSI/heatmap, stop and ask for timestamp synchronization details.

If there is no way to map heatmap cells or RF-derived blobs to ROI/world/user coordinates, stop and ask for calibration or geometry assumptions.

If candidate ROI information is unavailable, the RF residual heatmap can still be generated, but ROI feature extraction cannot be completed. In that case, implement only the residual heatmap and global intent features, then stop before ROI feature extraction.

### 1.3 Current user-provided inputs match expectation

The user said the available input is:

- 6DoF of user.
- Raw CSI data of the whole environment.
- Or completed heatmap from the existing heatmap generator.

This matches the expectation of this plan. Therefore, continue implementation.

### 3.1 Current pipeline

Current system described by user:

```text
Current 6DoF
    → Frustum / Occlusion Culling
    → ML model
    → ROI selection
```

Existing CSI code:

```text
Raw CSI
    → standard Doppler heatmap generation
```

### 3.2 Proposed pipeline

```text
Raw CSI
    → existing heatmap generator
    → standard heatmap H_rf

6DoF stream
    → head linear/angular velocity estimation
    → expected 6DoF-induced motion field

H_rf + expected 6DoF motion
    → residual Doppler intent heatmap H_res

Candidate ROIs from frustum / occlusion culling
    + H_res
    → RF-ROIAlign feature extraction
    → per-ROI feature vectors

Per-ROI feature vectors
    + original visual/geometric/6DoF features
    → ML model
    → ROI selection
```

### 3.3 Main contribution

Do not simply feed the raw Doppler heatmap into the ML model.

Instead, convert it into explainable features:

1. RF motion energy.
2. 6DoF-residual RF motion energy.
3. Motion-to-ROI alignment.
4. Time-to-contact.
5. Micro-Doppler bandwidth.
6. Doppler entropy.
7. Visibility-conflict score.
8. RF confidence.
9. Temporal growth.
10. Global RF intent direction.

## 5. Main algorithmic idea

### 5.1 Standard heatmap meaning

The existing generator produces a standard RF sensing heatmap. It answers:

> Where is RF Doppler / ToF / AoA energy strong?

This is useful but not directly optimized for ROI selection.

### 5.2 Residual heatmap meaning

RDITH should answer:

> Where is there RF-observed physical motion that is not already explained by the HMD's 6DoF, and how relevant is that motion to candidate ROIs?

### 5.3 Residual motion definition

Estimate head linear velocity:

```text
v_H(t) = [p_H(t) - p_H(t-dt)] / dt
```

Estimate head angular velocity:

```text
omega_H(t) = rotation_difference(R_H(t-dt), R_H(t)) / dt
```

For any RF cell or blob at position `x_world`, expected velocity from head/body rigid motion approximation:

```text
v_6dof(x,t) = v_H(t) + omega_H(t) × (x_world - p_H(t))
```

RF residual velocity:

```text
v_res(x,t) = v_RF(x,t) - v_6dof(x,t)
```

Residual score:

```text
S_residual(x,t) = P_motion(x,t) * ||v_res(x,t)||
```

If only scalar Doppler is available and full 3D velocity is not available, use projected residual Doppler:

```text
fd_expected = project_velocity_to_doppler(v_6dof, geometry)
fd_residual = fd_observed - fd_expected
S_residual = energy * abs(fd_residual)
```

### 7.7 `estimate_rf_velocity`

```python
def estimate_rf_velocity(
    active_cells: list,
    cell_map: dict,
    heatmap_result: dict,
    calibration: dict
) -> list:
    """
    Estimate RF-observed velocity for active cells or blobs.

    Inputs:
        active_cells:
            Output of extract_active_heatmap_cells().
        cell_map:
            Output of map_heatmap_cells_to_world().
        heatmap_result:
            Heatmap dictionary containing fd_axis and RF metadata.
        calibration:
            RF geometry, wavelength, center frequency, Tx/Rx positions.

    Outputs:
        rf_motion_cells:
            List of arrays or records with fields:
                position_world or pseudo_position
                velocity_world if estimable
                doppler_hz
                energy
                confidence

    Definition:
        Convert Doppler frequency into velocity information.
        If multiple links / AoA are available, estimate vector velocity.
        If only one scalar Doppler is available, preserve scalar projected velocity and mark vector velocity as unavailable.
    """
```

Formula:

```text
v_radial = fd * lambda / 2       # monostatic approximation
```

For bistatic or CSI links, use link-specific geometry if available:

```text
fd ≈ -(1 / lambda) * b_l(x)^T * v
b_l(x) = unit(Tx_l → x) + unit(Rx_l → x)
```


```text

residual_doppler = observed_doppler - expected_doppler
```

### 7.13 `compute_micro_doppler_bandwidth`

```python
def compute_micro_doppler_bandwidth(
    doppler_values_hz: np.ndarray,
    energies: np.ndarray
) -> float:
    """
    Compute weighted Doppler bandwidth.

    Inputs:
        doppler_values_hz:
            np.ndarray of Doppler frequencies.
        energies:
            np.ndarray of nonnegative energy values.

    Outputs:
        bandwidth_hz:
            Weighted standard deviation of Doppler frequencies.

    Definition:
        Higher bandwidth often indicates articulated or complex motion, such as hand/arm motion.
    """
```

Formula:

```text
p_i = energy_i / sum(energy)
mean = sum(p_i * fd_i)
bandwidth = sqrt(sum(p_i * (fd_i - mean)^2))
```

```python
    Inputs:

        energies:

    Definition:
        Doppler entropy measures motion complexity.
    """
```

Formula:

```text

entropy = -sum(p * log(p + eps))
```

```python
            List[RFBlob] near the ROI.
        pose_kinematics_at_t:
            Head kinematic dictionary for current frame.
        config:
            Feature parameters.

    Outputs:
        features: dict with keys:
            rf_motion_energy
            rf_residual_energy
            rf_velocity_x
            rf_velocity_y
            rf_velocity_z
            rf_approach_user
            rf_motion_to_roi_alignment
            rf_time_to_contact
            rf_micro_doppler_bandwidth
            rf_doppler_entropy
            rf_confidence
            rf_visibility_conflict
            rf_temporal_growth
            rf_blob_count
            rf_nearest_blob_distance

    Definition:
        These features are explainable ML inputs for ROI selection.
    """
```

Feature definitions:

#### `rf_motion_energy`

```text
sum(blob.energy for nearby blobs)
```

Meaning: how much RF motion is near the ROI.

#### `rf_residual_energy`

```text
sum(blob.residual_energy for nearby blobs)
```

Meaning: how much nearby motion is not explained by 6DoF.

#### `rf_velocity_x/y/z`

Energy-weighted average velocity:

```text
v_avg = sum(E_blob * v_blob) / sum(E_blob)
```

If vector velocity unavailable, set to NaN and expose scalar Doppler features instead.

#### `rf_approach_user`

```text
-head_direction_to_roi dot v_avg
```

or:

```text
-v_avg dot unit(roi.center_world - head_position)
```

Meaning: positive if motion approaches the user/headset.

#### `rf_motion_to_roi_alignment`

For blob centroid `c` and ROI center `o`:

```text
alignment = cosine(v_blob, o - c)
```

Energy-weighted over nearby blobs.

Meaning: high value means motion is moving toward the ROI.

#### `rf_time_to_contact`

```text
distance = norm(o - c)
closing_speed = max(0, dot(v_blob, unit(o - c)))
ttc = distance / (closing_speed + eps)
```

Use clipped or transformed version for ML:

```text
rf_ttc_score = exp(-ttc / tau)
```

#### `rf_micro_doppler_bandwidth`

Weighted average of blob Doppler bandwidth.

#### `rf_doppler_entropy`

Weighted average of blob Doppler entropy.

#### `rf_confidence`

Weighted average blob confidence.

#### `rf_visibility_conflict`

```text
rf_visibility_conflict = rf_residual_energy * (1 - roi.visibility)
```

Meaning: strong RF motion near an occluded or not-yet-visible ROI.

#### `rf_temporal_growth`

```text
rf_residual_energy(t) - rf_residual_energy(t-1)
```

Requires previous frame cache.

#### `rf_blob_count`

Number of RF blobs within ROI support radius.

#### `rf_nearest_blob_distance`

Minimum distance from ROI center to nearby RF blob centroid.

## 9. Example end-to-end API

Suggested top-level function:

```python
def run_rdith_pipeline(
    heatmap_path: Optional[str],
    raw_csi_path: Optional[str],
    pose_path: str,
    roi_path: Optional[str],
    calibration_path: str,
    config_path: str,
    output_path: str,
    heatmap_type: str = "AoA-ToF-Doppler"
) -> dict:
    """
    End-to-end RDITH feature extraction pipeline.

    If heatmap_path is provided, load completed heatmap.
    Else if raw_csi_path is provided, generate heatmap from CSI using existing code.
    Then align 6DoF, compute residual heatmap, extract RF blobs, pool ROI features, and export ML dataset.
    """
```

Pseudo-code:

```python
config = load_json(config_path)
calibration = load_json(calibration_path)
poses = load_poses(pose_path)

if heatmap_path is not None:
    heatmap = load_heatmap_result(heatmap_path, heatmap_type)
elif raw_csi_path is not None:
    csi = load_csi(raw_csi_path)
    heatmap = generate_standard_heatmap_from_csi(csi, config, heatmap_type)
else:
    raise ValueError("Need either heatmap_path or raw_csi_path")

candidate_rois = load_candidate_rois(roi_path) if roi_path else None
validate_rdith_inputs(heatmap, poses, candidate_rois, calibration)

aligned_poses = align_pose_to_heatmap_timestamps(poses, heatmap["timestamps"])
kinematics = compute_head_kinematics(aligned_poses)

active_cells = extract_active_heatmap_cells(heatmap)
cell_map = map_heatmap_cells_to_world(heatmap, calibration)
rf_motion_cells = estimate_rf_velocity(active_cells, cell_map, heatmap, calibration)
residual_cells = compute_residual_motion_cells(rf_motion_cells, kinematics, calibration)
residual_heatmap = build_residual_heatmap(heatmap, residual_cells, output_mode="sparse")

blobs = extract_rf_blobs(residual_heatmap)
tracked_blobs = track_rf_blobs(blobs)

global_features = compute_global_intent_features(tracked_blobs, kinematics, config["sector_config"])

if candidate_rois is not None:
    roi_rf_features = rf_roi_align(tracked_blobs, candidate_rois, kinematics, config)
    ml_dataset = merge_features_for_ml(candidate_rois, roi_rf_features, global_features)
    export_rdith_features(ml_dataset, output_path)
else:
    export_residual_heatmap(residual_heatmap, output_path)

return {
    "heatmap": heatmap,
    "residual_heatmap": residual_heatmap,
    "tracked_blobs": tracked_blobs,
    "global_features": global_features,
    "ml_dataset": ml_dataset if candidate_rois is not None else None,
}
```

## 10. Things that must be noticed

### 10.1 Axis order is dangerous

The existing code may flatten and reshape spectrum arrays. Before implementing RDITH, confirm the exact axis order for every heatmap type.

For `ToF-Doppler`, likely:

```text
frame × tau × fd
```

For `AoA-ToF-Doppler`, verify carefully. Do not assume blindly.

### 10.2 ToF-Doppler alone may not localize a ROI in 3D

If only ToF-Doppler heatmap is available, the RF cell tells delay and Doppler, but not full 3D position. ROI alignment may be weak or impossible without:

- AoA,
- multiple Tx/Rx links,
- known environment geometry,
- or additional calibration.

If the project needs spatial ROI pooling, prefer `AoA-ToF-Doppler` or multi-link geometry.

### 10.3 Timestamp synchronization is mandatory

Residual subtraction only makes sense if CSI/heatmap frames and 6DoF frames are aligned. Even 100 ms misalignment can corrupt intent features.

### 10.4 Coordinate calibration is mandatory for ROI features

To compute motion-to-ROI alignment or time-to-contact, the RF blobs and ROIs must be in the same coordinate frame.

Required calibration examples:

```text
RF coordinate frame → world coordinate frame
HMD tracking frame → world coordinate frame
Tx/Rx positions
antenna orientation
center frequency / wavelength
```

### 10.5 Do not overclaim full body intent

The residual heatmap estimates unexplained physical motion. It is not direct mind reading. It should be described as a physical motion-intent prior, not guaranteed user intention.

### 10.6 Residual subtraction may remove useful motion if model is too aggressive

If the 6DoF expected motion field is too broad, it may remove actual hand/body motion. Keep ablation options:

```text
raw RF energy
normal Doppler energy
6DoF residual energy
```

Do not delete raw features. Include both raw and residual features for the ML model.

### 10.7 RF multipath can create false blobs

Use confidence features:

- multi-link agreement,
- temporal persistence,
- spatial consistency,
- reasonable velocity range,
- heatmap peak sharpness.

### 10.8 Real-time performance

Avoid dense full 3D processing. Use:

- percentile / CFAR thresholding,
- top-K active cells,
- sparse blobs,
- per-ROI pooling.

### 10.9 Missing vector velocity

If only scalar Doppler is available, do not fake a 3D velocity vector. Use scalar projected Doppler features and mark vector velocity fields as unavailable / NaN.

### 10.10 Evaluation must include ablation

Required ablations:

```text
6DoF only
6DoF + frustum/occlusion
6DoF + raw heatmap features
6DoF + normal Doppler features
6DoF + RDITH residual features
```

Metrics:

```text
ROI prediction accuracy
Top-k ROI recall
future ROI IoU
time gained before ROI switch
rendering/bandwidth saved
false high-priority ROI ratio
feature attribution/explainability
```

## 12. What should not be done

Do not feed the raw heatmap directly into the ML model as the only RF input.

Do not remove the existing heatmap generator.

Do not assume ToF-Doppler gives exact 3D location.

Do not ignore timestamp alignment.

Do not mix coordinate frames silently.

Do not fabricate 3D velocity if only scalar Doppler exists.

Do not call the result "attention" without explaining that it is an RF-based physical motion prior.
